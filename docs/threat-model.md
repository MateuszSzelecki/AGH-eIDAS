# Threat Model & Privacy Analysis

This document outlines the security mechanisms implemented in the ZeroAge system, demonstrating compliance with the eIDAS 2.0 (EUDI ARF) requirements and mitigating key attack vectors. The architecture leveraging SD-JWT VC ensures high privacy and anti-tracking standards for users.

## 1. Replay Attacks
**Risk:** An attacker intercepts an authentication session or presentation payload and attempts to reuse it for their own gain.
**Mitigation:** 
- **OIA_02 (Cryptographic Key Binding):** eIDAS mandates that the presentation holds a cryptographic proof tying the verifiable credential to the user's specific secure hardware (Device Binding in SD-JWT VC). The credential cannot be extracted and replayed on an unauthorized device.
- **Nonce & Session Validation:** The OpenID4VP presentation flow securely binds a unique `nonce` generated by the Verifier, proving that the presentation is freshly generated for this specific session and completely invalidating the duplicated payload.

## 2. Credential Sharing
**Risk:** A legitimate adult user deliberately shares their age credential with an unauthorized minor to allow them to bypass mandatory age verification (e.g., for purchasing 18+ items).
**Mitigation:**
- **WUA_03 / WUA_09 (Wallet Unit Attestation):** The system cryptographically confirms that the keys used to sign the age proof are stored within the smartphone's Secure Enclave, making it impossible to "copy" the credential to another device.
- **WIAM_01 (Strong Authentication):** Local wallet authentication (FaceID, Fingerprint, or secure PIN) tied to LoA High standards ensures that only the rightful owner can trigger the age-proof generation. Sharing the credential requires physically handing over the locked device and the biometric/PIN access.

## 3. Issuer/Verifier Collusion
**Risk:** The Issuer and the Verifier conspire to correlate the identity of the user across multiple verifications, tracking where and when the user authenticates, thus breaking anonymity.
**Mitigation:**
- **ISSU_37 & ISSU_43 (Batch Issuance & Once-only Attestations):** Instead of using a single verifiable credential with a static identifier across all transactions, the Wallet requests a batch of credentials covering identical claims from the Issuer. During verification, each credential is mathematically "Once-Only" (Method A), meaning it is selectively disclosed and permanently discarded after a single presentation.
- **OIA_16 (Anti-tracking Requirements):** Verifying endpoints in the system are legally restricted from keeping unique tracking identifiers long-term, enforcing the Zero-Knowledge principle.

## 4. Privacy Guarantees (Anonymous Age Verification)
**Risk:** Relying Parties extract, learn, or deduce more personal data than absolutely necessary (e.g., exact birthdate) from the user during age-gated verification.
**Mitigation:**
- **ZKP-Based Age Verification:** Instead of sharing the `birthdate` attribute, the ZeroAge system utilizes **Zero-Knowledge Proofs (Circom circuits)** to prove that the user was born before a specific date (i.e., they are 18+). The Verifier receives only a cryptographic proof of the predicate, never the underlying date, preventing user profiling based on age or birthday.
- **Minimization with SD-JWT:** For attributes that don't require ZKPs, **SD-JWT** allows the Wallet to surgically reveal only necessary claims (e.g., "Student Status: Active") while keeping all other identity data salted and hidden from the Verifier.
- **OIA_05 / OIA_06 (Explicit Consent):** The EUDI Wallet interface forces explicit, fully informed user approval explaining precisely who requested the data and exactly which attributes will be transmitted (e.g., "The verifier will only see a proof that you are over 18") before sending any presentation.
- **UD_01 (User Dashboard Transparency):** The Wallet strictly maintains a local log capturing the history of age verifications, giving the user a persistent overview of which services have verified their age.
